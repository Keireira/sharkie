# Backend

The backend is a Rust API server built with Axum 0.8 and SQLx 0.8, running on Tokio.

## Module Structure

```
src/
├── main.rs          # Entry point: bootstrap, middleware stack, graceful shutdown
├── config.rs        # Config struct loaded from environment variables
├── error.rs         # Unified ApiError enum with IntoResponse + error counters
├── state.rs         # AppState: DB pool, config, HTTP client
├── models.rs        # DB models, API DTOs, OpenAPI schemas (utoipa)
├── telemetry.rs     # Prometheus metrics setup, HTTP middleware, pool monitor
├── routes/
│   ├── mod.rs       # Router assembly, OpenAPI doc, compression layer
│   ├── health.rs    # GET /health (DB connectivity check)
│   ├── currencies.rs # GET /currencies
│   └── history.rs   # GET /history (date parsing, TZ clamping, filtering, rebasing, pagination)
└── services/
    ├── mod.rs
    └── rates.rs     # Backfill, daily fetch, scheduler, batch insert
```

## Architecture Pattern

The codebase follows the **routes → services → db** pattern:

1. **Routes** handle HTTP concerns (parsing query params, returning JSON, cache headers)
2. **Services** contain business logic (backfill, scheduling, rate calculations)
3. **Database** access is done through SQLx with parameterized queries (SQL injection safe)

## Configuration

All configuration is via environment variables:

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `DATABASE_URL` | yes | — | PostgreSQL connection string |
| `CURRENCY_API_URL` | no | `https://api.currencyapi.com/v3/` | External rate API base URL |
| `CURRENCY_API_KEY` | no | `""` | API key for rate fetching |
| `PORT` | no | `3000` | Server listen port |
| `ALLOWED_ORIGINS` | no | `*` (any) | Comma-separated CORS origins |
| `RUST_LOG` | no | `sharkie=info,tower_http=info` | Log level filter |

## Error Handling

A unified `ApiError` enum covers all error cases:

- `NotFound` → 404
- `InvalidInput(String)` → 400
- `PayloadTooLarge` → 413
- `Internal(String)` → 500 (message logged server-side, generic response to client)

Automatic `From` implementations for `sqlx::Error` and `reqwest::Error` allow using `?` in handlers.

## Middleware Stack

Middleware is applied in `main.rs` as a layered tower stack. Order (outermost first):

| Layer | Crate | Purpose |
|-------|-------|---------|
| **Prometheus metrics** | `metrics` | Request count + latency histograms (outermost — captures everything) |
| **Timeout** | `tower-http::timeout` | 30s per-request deadline → 408 |
| **Body limit** | `tower-http::limit` | 256 Kb request body cap |
| **Rate limiting** | `tower_governor` | 60 req/min per IP (SmartIpKeyExtractor) |
| **Request ID** | `tower-http::request_id` | UUID v4 in `x-request-id`, propagated to response |
| **Tracing** | `tower-http::trace` | Structured logs with method, URI, request ID |
| **Security headers** | `tower-http::set_header` | HSTS, nosniff, DENY, referrer-policy |
| **CORS** | `tower-http::cors` | Configurable via `ALLOWED_ORIGINS` |
| **Compression** | `tower-http::compression` | gzip + brotli (in router) |

### Security Headers

Every response includes:

```
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
```

### Rate Limiting

60 requests per minute per client IP via `tower_governor`. Uses `SmartIpKeyExtractor` which reads `X-Forwarded-For` / `X-Real-IP` headers (from Cloudflare/Nginx) and falls back to peer IP for direct connections. Returns `429 Too Many Requests` with `x-ratelimit-limit`, `x-ratelimit-remaining`, and `retry-after` headers.

### Request Tracing

Every request gets a UUID (`x-request-id`) that appears in:
- Response header (for client-side correlation)
- All log lines within the request span
- Structured fields: `method`, `uri`, `request_id`

The `/history` handler additionally logs `query_ms` (DB query time), `total_ms` (full handler time), `days` (result count), and `includes_today`.

## Cache Headers

Per-route `Cache-Control` for Cloudflare (CDN) + Nginx + browser caching:

| Endpoint | Cache-Control | Rationale |
|----------|--------------|-----------|
| `/health` | `no-store` | Always fresh for monitoring |
| `/currencies` | `public, s-maxage=3600, max-age=300` | Rarely changes (CDN 1h, browser 5m) |
| `/history` (past only) | `public, s-maxage=86400, max-age=86400, immutable` | Historical data never changes |
| `/history` (includes today) | `public, s-maxage=900, max-age=300` | Today's data refreshes every 4h (CDN 15m, browser 5m) |

## Response Compression

All responses are compressed with gzip or brotli via `tower-http::compression`, negotiated by the client's `Accept-Encoding` header. Exchange rate JSON compresses ~80%.

## Timezone-Aware Date Handling

Clients in timezones ahead of UTC (e.g., GMT+10) may request dates that are "today" for them but "tomorrow" for the server. To prevent 404 errors, the server clamps all future dates to today (UTC). This ensures the latest available data is always returned.

## Database Connection Resilience

On startup, the backend retries the database connection with exponential backoff (1s, 2s, 4s, 8s, 16s — up to 5 attempts) before panicking. The pool also has a 5-second `acquire_timeout`.

## Background Tasks

### Pool Monitor (`telemetry::pool_monitor`)
Runs every 30 seconds. Executes `SELECT 1` as a health probe and exports DB pool gauges (`db_pool_size`, `db_pool_idle`, `db_pool_active`, `db_pool_healthy`) to Prometheus. Logs warnings on failure and info when connectivity is restored.

### Backfill (`backfill_missing_rates`)
Runs on startup. Iterates from 2000-01-01 to yesterday and fetches any missing dates from the external API. Exports `backfill_remaining_dates` gauge and `backfill_dates_total` counter to Prometheus.

### Daily Updater (`daily_rate_updater`)
Background Tokio task that:
1. Checks if today's rates exist on startup; fetches if missing
2. Sleeps until the next scheduled UTC time
3. Schedule: 00:05, 04:00, 08:00, 12:00, 16:00, 20:00, 23:55

Increments `rate_fetch_failures_total` counter on failure.

### Batch Insert
Both backfill and daily fetch use a single parameterized `INSERT ... ON CONFLICT DO UPDATE` query for efficiency.

## Monitoring

Prometheus metrics are exposed at `GET /metrics` via the `metrics` + `metrics-exporter-prometheus` crates. The HTTP metrics middleware is the outermost layer, capturing all requests including rate-limited and timed-out ones. Path labels are normalized to known routes to prevent cardinality explosion.

Alert rules are defined in `prometheus/alerts.yml` covering: 5xx spikes, DB pool exhaustion, backfill failures/stalls, rate limit spikes, and rate fetch failures.

## OpenAPI

The API spec is generated at compile time via `utoipa` and served at `GET /openapi.json`. All route handlers are annotated with `#[utoipa::path]`, and models derive `ToSchema` / `IntoParams`.

## Database

Single table with composite primary key:

```sql
CREATE TABLE exchange_rates (
    currency_code VARCHAR(5) NOT NULL,
    date DATE NOT NULL,
    rate NUMERIC(20, 10) NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (currency_code, date)
);

CREATE INDEX idx_rates_date ON exchange_rates(date);
```

Migrations are managed by SQLx and run automatically on startup.
