# Security

Security guidelines are documented in `rules/frontend-security.md` and `rules/hack.md`. This page summarizes the key measures in place.

## Backend Security

### SQL Injection Prevention
All database queries use parameterized placeholders (`$1`, `$2`, ...) via SQLx. No string interpolation is used for user-provided values in SQL.

### Input Validation
- Date parameters are parsed with strict `YYYY-MM-DD` format
- Currency codes are uppercased and filtered
- Empty inputs return 400 errors with descriptive messages

### Error Responses
Internal errors are logged server-side but return generic messages to clients. Stack traces and internal details are never exposed.

### CORS
Configurable via `ALLOWED_ORIGINS` environment variable. Defaults to permissive (`Any`) for development; should be restricted in production.

### Graceful Shutdown
The server handles SIGTERM and Ctrl+C signals, allowing in-flight requests to complete.

### Response Size Limits
API responses are capped at 512 KB to prevent memory abuse from overly broad queries.

## Frontend Security

### No Server Runtime
The frontend is statically exported — there is no Node.js server in production, eliminating an entire class of server-side vulnerabilities.

### API Proxying
All API calls go through Nginx's `/api/proxy/` reverse proxy, hiding the backend's direct address from clients.

### Content Security
- JSON-LD for structured data uses `dangerouslySetInnerHTML` only for static, developer-controlled data (not user input)
- External links should use `rel="noopener noreferrer"`
- No user-generated HTML is rendered

### Environment Variables
- `NEXT_PUBLIC_*` variables are the only ones bundled into the client
- Sensitive keys (`DATABASE_URL`, `CURRENCY_API_KEY`) are server-side only

## Recommendations

Per the project's security rules, these should be implemented for a production deployment:

- [ ] Rate limiting at the reverse proxy level (nginx `limit_req`)
- [ ] Security headers (HSTS, X-Content-Type-Options, X-Frame-Options)
- [ ] Restrict CORS origins to specific domains
- [ ] Enable `cargo audit` in CI
- [ ] Run `pnpm audit` in CI
- [ ] Add Content-Security-Policy headers
- [ ] Set up monitoring for 4xx/5xx spikes
