# Architecture

Sharkie follows a classic three-tier architecture: a Rust API backend, a PostgreSQL database, and a statically-exported React frontend served by Nginx.


The frontend proxies API requests through Nginx at `/api/proxy/` to the Rust backend.

## Data Flow

### Rate Fetching

1. On startup, the backend runs `backfill_missing_rates()` to fill any gaps from 2000-01-01 to yesterday
2. A background task (`daily_rate_updater`) fetches current-day rates at scheduled UTC times: 00:05, 04:00, 08:00, 12:00, 16:00, 20:00, 23:55
3. All rates are stored as USD-based in the `exchange_rates` table
4. The API recalculates rates on-the-fly when a non-USD base is requested

### Frontend Data Management

1. `useRatesQuery` fetches rates for the selected period and currencies
2. `useFeedRateStore` normalizes all fetched data to USD and caches it in React Query
3. `useRate` provides single-pair conversion by checking the centralized cache first, fetching only if needed
4. `useRangeRates` provides date-range data for charts and tables

## Key Design Decisions

- **Static export** — the frontend is built as static HTML/JS and served by Nginx, avoiding Node.js in production
- **USD normalization** — all rates are stored relative to USD; conversion to other bases is computed at query time
- **Single reqwest::Client** — shared HTTP client across the application (not created per request)
- **Graceful shutdown** — the server handles SIGTERM/Ctrl+C cleanly
- **CORS** — configurable via `ALLOWED_ORIGINS` environment variable
- **Tracing** — structured logging with `tracing` crate, configurable via `RUST_LOG`
